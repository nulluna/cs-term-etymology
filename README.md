# cs-term-etymology

一个面向**计算机术语解析与词源溯源**的开源 Claude/ChatGPT Skill。

## 快速安装

```bash
npx skills add nulluna/cs-term-etymology
```

当前仓库按**单 skill 仓库**组织，根目录提供 `SKILL.md`，可直接按仓库安装。

它不只是做“术语翻译”或“词典释义”，而是把一个计算机名词拆成四层能力来回答：

1. **词源层**：这个词原来来自哪里，原义是什么，如何演变到今天的形式
2. **定义层**：它在计算机领域里具体指什么，属于哪个子领域
3. **术语映射层**：标准中文 / 标准英文对应是什么，哪些译法更常见或更标准
4. **辨析层**：它和相近术语到底差在哪里，哪些中文译法容易误导

适用领域尤其包括：

- 操作系统
- 编译原理 / 语言实现
- 网络
- 软件工程
- 计算机基础

## 适合解决的问题

- “kernel 为什么翻译成内核？”
- “daemon 这个词在 Unix 里是怎么来的？”
- “thread / process / task 到底有什么区别？”
- “守护进程、线程、套接字这些词对应的标准英文是什么？”
- “帮我从一段技术文章里抽取关键术语并逐项解释。”
- “请重点讲词源 / 重点讲中文译法 / 重点讲操作系统语境。”

## 设计特点

### 1. 面向计算机术语，不做泛语言学扩张

这个 skill 专门处理计算机术语，不试图覆盖所有自然语言词汇。

### 2. 强调“为什么这样叫”

输出不止给定义，还会说明：

- 原义与计算机义之间的联系
- 是否发生了隐喻、转义、缩窄、扩大
- 中文译名是否保留了原词的语义核心

### 3. 显式处理中英双语检索

该 skill **依赖 `exa-search` skill** 做网页检索。

由于 Exa 从 2025 年起会默认按**查询语言**做语言过滤，所以如果既要拿：

- 英文词源页（Wiktionary、Etymonline、Wikipedia 英文页）
- 中文术语页（glossary、术语库、官方中文翻译资料）

就不能只发一个中英混合查询，而要显式拆成两轮：

- **英文查询**：负责词源与英文定义
- **中文查询**：负责中文术语映射与译名判断

这样可以避免结果被单一语言来源“带偏”。

## 依赖

- [exa-search](https://github.com/nulluna/claude-context-dae-skills/tree/main/exa-search) skill

推荐来源链路：

- **词源层**：Wiktionary、Etymonline
- **定义层**：Wikipedia、官方文档、RFC、经典技术资料
- **术语映射层**：公开 glossary、术语库、官方翻译资料
- **辨析层**：由 skill 自身模板与推理规则完成

## 输出能力

### 单术语

默认可输出：

1. 术语原文
2. 标准中英文对应
3. 一句话定义
4. 详细技术解释
5. 词源与构词分析
6. 进入计算机领域后的语义迁移
7. 中文译名分析
8. 相近术语辨析
9. 参考来源

### 多术语对比

支持先给总览表，再做逐项解释，最后总结关键边界。

### 正文抽取

支持从技术文章、笔记、RFC 片段中抽取关键术语，再按重要性展开解释。

## Release 发布建议

为了方便版本管理与公开分发，建议创建 Git tag 与 GitHub Release；但对 `skills.sh` 安装来说，**release 附件不是必需条件**。

推荐最小做法：

1. 使用清晰的版本号打 tag，例如：`v0.1.0`
2. 基于该 tag 创建 GitHub Release
3. 在 release note 中简要说明本版本的新增、修正与兼容性变化

如果你希望额外提供打包产物用于归档或镜像分发，也可以把下面文件作为可选附件上传：

- `cs-term-etymology.skill`
- `cs-term-etymology.zip`

但它们更适合作为补充产物，而不是 `skills.sh` 安装的前提。

### release note 建议

可采用下面这样的结构：

- 本版本新增 / 调整了哪些能力
- 是否调整了触发描述或 examples
- 是否更新了参考资料或输出模板
- 用户安装方式是否有变化

### 面向 `skills.sh` 的分发说明

- 当前仓库保持单 skill 结构

## 示例

### 示例 1

**输入：**

```text
kernel 这个词为什么会变成操作系统里的“内核”？
```

**期望输出重点：**

- kernel 的一般英语原义
- 它与“核心部分”语义的关系
- 进入操作系统语境后的专业化含义
- “内核”为什么是稳定译法
- 与 `core` 的差异

### 示例 2

**输入：**

```text
解释 process / thread / task 的区别，重点讲中文误译问题
```

**期望输出重点：**

- 三者在操作系统语境中的边界
- `task` 在不同系统/资料中的不稳定性
- “进程 / 线程 / 任务”为什么容易混淆

### 示例 3

**输入：**

```text
下面这段文章里有哪些关键计算机术语？请抽取后逐项解释，并更关注词源。
```

**期望输出重点：**

- 术语抽取
- 子领域识别
- 词源优先的解释顺序

## 适合作为开源 skill 的原因

- **边界清晰**：只做计算机术语
- **输出稳定**：有独立 research / output references
- **来源可复核**：强调来源分层与不确定性标注
- **适配中英混合技术语境**：非常适合中文开发者学习计算机术语
